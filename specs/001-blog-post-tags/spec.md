# Feature Specification: Blog post tags and tag pages

**Feature Branch**: `001-blog-post-tags`  
**Created**: 2026-04-06  
**Status**: Draft  
**Input**: User description: "博客文章支持 tags；每篇至少一个、可多个；通过 `/tags/spec/` 这类路由查看含该 tag 的全部文章，便于按类浏览。"

## Clarifications

### Session 2026-04-06

- Q: 从未在任何文章元数据中出现过的 tag slug，其 `/tags/{slug}/` 应与「有该标签但当前 0 篇文章」如何区分？ → A: **不区分**；均视为可访问的标签聚合页，HTTP 成功响应，展示空列表与简短说明（选项 A）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Browse posts by tag (Priority: P1)

读者在浏览器中打开某一标签的专属地址（路径形如 `/tags/{标签}/`，示例 `/tags/spec/`），应看到**所有**带有该标签的文章列表，顺序与站点主列表一致（默认按发布日期，新的在前）。

**Why this priority**: 这是按主题筛选内容的核心价值；没有它，标签数据无法服务读者。

**Independent Test**: 只实现标签页与数据关联即可演示：给定几篇带标签的示例文章，打开对应标签地址，核对列表内容与标签匹配情况。

**Acceptance Scenarios**:

1. **Given** 至少两篇文章带有标签 `spec`，**When** 读者打开该标签的专属地址，**Then** 列表中恰好包含这两篇且不包含未带 `spec` 的文章。
2. **Given** 一篇文章带有多个标签，**When** 读者分别打开其中每个标签的专属地址，**Then** 该文章在每个相关标签列表中都会出现。

---

### User Story 2 - Discover tags from an article (Priority: P2)

读者在阅读单篇文章时，能看到该文所属的全部标签，并能通过一次点击（或同等单次操作）跳转到对应标签的专属列表页。

**Why this priority**: 提升发现与站内跳转；在 P1 已有标签页的前提下，把标签从“隐藏元数据”变成可导航入口。

**Independent Test**: 在单篇文章页展示可点击的标签链接；点击后落地页与 P1 的标签列表行为一致。

**Acceptance Scenarios**:

1. **Given** 一篇文章带有标签 `spec` 与 `tutorial`，**When** 读者在文章页点击 `spec`，**Then** 进入 `spec` 的标签列表页且列表包含当前文章。
2. **Given** 文章仅有一个标签，**When** 读者点击该标签，**Then** 进入对应标签列表页。

---

### User Story 3 - Every post declares tags (Priority: P3)

内容作者在撰写文章时，必须为每篇稿件指定**至少一个**标签；支持同一篇文章挂多个标签。站点在构建或校验流程中应能发现“零标签”的稿件并阻止其按正常已发布文章处理（或按项目约定的等价门禁）。

**Why this priority**: 保证数据模型与用户承诺一致，避免标签页与列表出现“无标签正文”的灰色地带。

**Independent Test**: 新增一篇仅含正文的稿件、不填标签，应触发校验失败或明确降级策略（与 Assumptions 中迁移策略一致）。

**Acceptance Scenarios**:

1. **Given** 一篇新文章元数据中包含至少一个标签，**When** 发布流程完成，**Then** 该文章出现在对应标签列表及主文章列表中。
2. **Given** 一篇新文章元数据中没有任何标签，**When** 执行发布/构建校验，**Then** 流程失败或按文档化的例外策略处理，且不会作为合规已发布文章出现在读者列表中。

---

### Edge Cases

- 某标签下**没有任何已发布文章**（包括：该 slug 从未在元数据中出现，或曾出现但当前无文）：标签专属地址仍 MUST 可访问，HTTP 成功语义，展示**空列表**并配有简短说明；**不**对「未知 slug」单独返回 404 或全局错误页。
- 同一篇文章元数据中重复声明同一标签：列表中对该文只出现一次。
- 标签在 URL 中的写法与元数据中的写法：采用**单一规范化规则**（大小写、空格与特殊字符），避免同一语义对应多个不同地址；具体规则见 Assumptions。
- 存量文章在功能上线前无标签：需一次性补齐标签或单独标记为草稿/排除在发布集之外（见 Assumptions）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 每篇面向读者发布的文章 MUST 在元数据中声明**至少一个**标签。
- **FR-002**: 每篇文章 MAY 声明多个标签；带有多标签的文章 MUST 出现在其**每一个**标签对应的列表中。
- **FR-003**: 系统 MUST 提供标签聚合页，其公开 URL 路径模式为 `/tags/{标签}/`（允许与常见实践一致的尾随斜杠变体，但应对外保持单一规范链接）。
- **FR-004**: 标签聚合页 MUST 只展示带有该标签的已发布文章，默认排序与站点主文章列表一致（日期降序），除非本规格另有说明。
- **FR-005**: 单篇文章阅读视图 MUST 展示该文全部标签，且每个标签 MUST 可导航至对应标签聚合页。
- **FR-006**: 博客正文源文件仍 MUST 存放在仓库根目录的 `posts/` 下，标签仅作为元数据扩展，不改变文章存放位置（与项目宪章一致）。
- **FR-007**: 标签的公开标识（用于 URL 与比对）MUST 遵循 Assumptions 中的规范化规则，确保同一标签不会分裂成多个互不关联的地址。
- **FR-008**: 任意符合路径模式的标签聚合 URL（规范化后的 `/tags/{slug}/`）MUST 以**可浏览的成功页面**呈现；当不存在带有该 slug 的已发布文章时，列表为空并附简短说明。**禁止**仅因「slug 从未在仓库元数据中出现」而返回未找到类错误响应。

### Key Entities *(include if feature involves data)*

- **标签 (Tag)**：一个可读的分类名或 slug；与 URL 片段一一对应（规范化后）；与多篇文章多对多关联。
- **文章 (Post)**：现有文章实体，增加**非空**标签集合；仍包含标题、日期、摘要等已有元数据。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 对任意已发布文章上出现的每个标签，读者都能在**一次导航**内从该文章到达仅包含该标签下文章的聚合页。
- **SC-002**: 对任意标签聚合页，列表中的文章**全部**带有该标签，且**不会遗漏**任何已发布且带有该标签的文章。
- **SC-003**: 随机抽查已发布文章样本（例如连续 10 篇），**100%** 带有至少一个标签。
- **SC-004**: 在带有 3 个标签的单篇文章上，读者从文章页可分别到达 3 个不同的标签聚合页，且每个页面都包含该文章。

## Assumptions

- 标签在 URL 中使用**小写、连字符分隔的 slug**（元数据可写中文或短语，发布管线将其映射为 slug），以减少重复与编码问题；若保留非 ASCII slug，则 MUST 统一采用 UTF-8 路径编码且全站规则一致。
- 功能交付时，**所有**现有 `posts/` 下已发布文章会补齐至少一个标签，或从“已发布集”中排除直至补齐（由实现计划在迁移步骤中写明）。
- 不提供独立“全标签云/索引页”除非后续单独规格要求；本规格仅要求**按标签浏览**与**从文内跳转**。
- 标签由作者维护，无读者自定义标签或社交标签审核流程。
- 读者输入的 slug **无需**事先存在于任何文章中：行为与「该标签下暂无文章」一致（见 Clarifications 2026-04-06），以降低路由与验收分支数量。
